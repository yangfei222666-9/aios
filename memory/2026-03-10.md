### 第二步：GitHub 学习 - Trigger 相关 ✅

**存档 ID:** t6q4hn  
**状态:** confirmed / design-only  
**定性:** 太极OS 有效设计资产

**研究项目：**
- semantic-router（语义路由）
- Parlant（上下文工程框架）

**4 个核心问题的答案：**

#### 1. 触发条件是怎么表达的？

**semantic-router 模式：**
- utterances（示例语句）
- semantic vector space（语义向量空间匹配）
- threshold（相似度阈值）

**Parlant 模式：**
- condition（自然语言条件）
- action（指令）
- priority（优先级）
- dependencies（依赖）
- exclusions（排斥）

**核心差异：**
- semantic-router：基于语义相似度（embedding + cosine similarity）
- Parlant：基于 LLM 判断（每轮动态评估条件是否成立）

#### 2. 触发判断发生在什么时候？

**semantic-router：**
- 每次用户输入时，在 LLM 生成前做路由决策
- 超快速（10ms 级别）
- 单次判断

**Parlant：**
- 每个对话轮次开始时，构建上下文前先做匹配
- 多阶段匹配（可以在 tool 调用后再次匹配）
- 动态上下文构建

**关键洞察：**
- semantic-router 是"路由层"，在 LLM 前做决策
- Parlant 是"上下文工程层"，在每轮动态过滤上下文

#### 3. 多个候选同时命中时怎么处理？

**semantic-router：**
- 返回最高分
- threshold 过滤
- 没有排斥/依赖

**Parlant：**
- priority 排序
- exclusion 排斥
- dependency 依赖
- 并行激活
- 冲突解决

**关键差异：**
- semantic-router：单选（one-hot）
- Parlant：多选 + 关系管理（multi-hot + relationships）

#### 4. 触发结果有没有证据留痕？

**semantic-router：**
- 没有内置留痕
- 可以扩展

**Parlant：**
- 完整 OpenTelemetry 追踪
- explainability 一等公民
- 记录每个 guideline 匹配、为什么被匹配、哪些被排斥、哪些 tool 被调用、最终上下文包含什么

**关键洞察：**
- Parlant 的 explainability 是生产级的，不是事后补的
- 每个决策都可以回溯和审计

### 今日输出（3 件）

#### 1. 架构启发

**Trigger 不是单一机制，而是分层的：**

- **Layer 1: 路由层（semantic-router）** - 快速决策，语义匹配，10ms 级别
- **Layer 2: 上下文工程层（Parlant）** - 动态过滤，关系管理，LLM 判断
- **Layer 3: 工具调用层** - 只有 observation 成立时才调用 tool
- **Layer 4: 证据留痕层** - 完整审计和回溯

**核心原则：**
> "Adding more rules makes the agent smarter, not more confused — because the engine filters context relevance, not the LLM."

#### 2. 与太极OS 的差距判断

**太极OS 当前状态：**
- ✅ 有 Agent 注册和调度
- ✅ 有 Skill 注册
- ❌ **没有 Skill 触发条件定义**
- ❌ **没有动态上下文匹配**
- ❌ **没有触发证据留痕**

**差距：**
1. 触发条件表达 - 太极OS 还没有 `skill_trigger_spec`，Skill 是全量加载的
2. 关系管理 - 没有 priority / dependency / exclusion
3. 证据留痕 - 没有记录"为什么激活"/"为什么没激活"
4. 多阶段匹配 - 没有 tool 调用后再次匹配的能力

**优势：**
- 太极OS 有 Reality Ledger（完整生命周期记录）
- 太极OS 有 Evolution Score（真实收益衡量）
- 太极OS 有易经状态引擎（状态判断框架）

**结论：**
太极OS 的底层能力（记录、评估、状态）比 Parlant 更完整，但 **Trigger 层缺失**。

#### 3. 可执行改进建议

**在 Skill 自动创建 MVP 中引入 `skill_trigger_spec`：**

```python
class SkillTrigger:
    """Skill 触发条件定义"""
    
    # Layer 1: 快速过滤（semantic-router 风格）
    activation_signals: List[str]      # 示例语句，用于 embedding 匹配
    negative_conditions: List[str]     # 排除条件
    required_context_keys: List[str]   # 必需的上下文字段（硬门槛）
    
    # Layer 2: 精确判断（Parlant 风格）
    condition: str                     # 自然语言条件，LLM 判断
    priority: int                      # 优先级（0-100）
    
    # Layer 3: 关系管理
    dependencies: List[str]            # 依赖的 observation/skill
    exclusions: List[str]              # 排斥的 skill
    
    # Layer 4: 证据留痕
    def should_activate(self, context: dict) -> TriggerDecision:
        """判断是否应该激活，返回决策 + 原因"""
        pass
```

**TriggerDecision 留痕结构：**

```python
class TriggerDecision:
    """触发决策结果"""
    
    should_activate: bool              # 是否激活
    score: float                       # 匹配分数（0-1）
    matched_signals: List[str]         # 命中的信号
    blocked_by: List[str]              # 被什么阻止
    reason: str                        # 判断原因
```

**实施路径：**

1. **Phase 1（当前 MVP）：** 只做 `activation_signals` + `negative_conditions` + `required_context_keys`
2. **Phase 2（v1.2）：** 增加 `condition` + `priority`
3. **Phase 3（v2.0）：** 增加 `dependencies` + `exclusions`
4. **Phase 4（v2.1）：** 完整证据留痕

**为什么重要：**
- 没有触发条件的 Skill 不能规模化
- 触发条件是 Skill 自动创建与按需激活之间的桥梁
- 证据留痕是 Self-Improving Loop 的基础
