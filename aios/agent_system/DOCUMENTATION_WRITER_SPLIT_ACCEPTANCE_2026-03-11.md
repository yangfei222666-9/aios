## 1. 拆分结果

### 原 Agent（已移除）

**Documentation_Writer**
- 角色：文档撰写员
- 任务：4 个大型任务（统一文档、demo、快速开始、FAQ）
- 调度：frequent (每 24h)
- 问题：任务过大，导致 coder agent 超时（60s）

### 新 Agent（4 个）

#### 1. Docs_Unified_Writer
- **角色：** 主文档统一员
- **目标：** 统一 AIOS 主文档到 README.md
- **任务范围：**
  - 本次只处理：README.md
  - 只补内容：合并 INSTALL.md、API.md、TUTORIAL.md
  - 输出文件：README.md
  - 不允许：修改其他文档
  - 完成标准：README.md 包含安装、API、教程三部分
- **调度：** weekly (每 168h)
- **优先级：** high

#### 2. Docs_Demo_Writer
- **角色：** Demo 文档撰写员
- **目标：** 撰写真实场景 demo 文档
- **任务范围：**
  - 本次只处理：docs/examples/ 目录
  - 只补内容：文件监控、API 健康检查、日志分析三个场景
  - 输出文件：docs/examples/*.md
  - 不允许：修改 README.md 或其他主文档
  - 完成标准：每个 demo 包含场景、代码、步骤、输出
- **调度：** weekly (每 168h)
- **优先级：** high

#### 3. Docs_QuickStart_Writer
- **角色：** 快速开始指南撰写员
- **目标：** 撰写 5 分钟快速开始指南
- **任务范围：**
  - 本次只处理：docs/QUICKSTART.md
  - 只补内容：5 分钟内跑起来第一个任务
  - 输出文件：docs/QUICKSTART.md
  - 不允许：修改 README.md 或其他文档
  - 完成标准：包含安装、配置、运行、验证四步，每步不超过 3 行命令
- **调度：** weekly (每 168h)
- **优先级：** high

#### 4. Docs_FAQ_Writer
- **角色：** FAQ 维护员
- **目标：** 维护常见问题解答文档
- **任务范围：**
  - 本次只处理：docs/FAQ.md
  - 只补内容：从 GitHub Issue、Telegram 提取的常见问题
  - 输出文件：docs/FAQ.md
  - 不允许：修改 README.md 或其他文档
  - 完成标准：至少 10 个问题，每个有清晰解答和示例
- **调度：** weekly (每 168h)
- **优先级：** normal

## 2. 关键改进

### 2.1 任务边界明确

**之前：**
```
"更新 AIOS 文档，补充新功能说明和使用示例"
```
- 过于宽泛
- 没有明确范围
- coder 不知道该做什么

**之后：**
```
"本次只处理：README.md"
"只补内容：合并 INSTALL.md、API.md、TUTORIAL.md"
"输出文件：README.md"
"不允许：修改其他文档"
"完成标准：README.md 包含安装、API、教程三部分"
```
- 明确输入输出
- 明确边界
- 明确完成标准

### 2.2 Prompt 结构收紧

每个子任务的 tasks 列表都遵循固定结构：
1. 本次只处理哪一个文档
2. 只补哪一类内容
3. 输出文件路径
4. 不允许顺带修改其他文档
5. 完成标准是什么

### 2.3 调度频率调整

**之前：**
- Documentation_Writer: frequent (每 24h)
- 每天都可能触发，增加超时风险

**之后：**
- 所有文档 Agent: weekly (每 168h)
- 降低触发频率，减少超时风险
- 文档更新不需要每天执行

## 3. Timeout 策略

### 当前设置

**保持 60s 不变**

**理由：**
1. 拆分后每个任务都更小
2. 先观察是否自然过关
3. 只有仍超时才考虑单独升到 90s/120s

### 后续观察

如果拆分后仍超时，再考虑：
- 对文档类单独升到 90s/120s
- 不一刀切放大所有任务 timeout

## 4. 验收标准检查

### ✅ 1. 每个文档子任务都能单独执行

- Docs_Unified_Writer: 只处理 README.md
- Docs_Demo_Writer: 只处理 docs/examples/
- Docs_QuickStart_Writer: 只处理 docs/QUICKSTART.md
- Docs_FAQ_Writer: 只处理 docs/FAQ.md

**结论：** 每个任务都有明确的输入输出，可以独立执行

### ✅ 2. 60s 内通过率明显提升（待观察）

**当前状态：** 已拆分，等待下次定时任务触发

**预期：**
- 每个子任务的工作量远小于原任务
- 60s 应该足够完成单个文档的更新

**观察方式：**
- 等待下次 weekly 触发（7 天后）
- 或手动触发一次 smoke test

### ✅ 3. 不再出现"大 prompt 一次包 4 件事"

**之前：**
```
tasks: [
  "统一文档到 README.md（合并 INSTALL/API/TUTORIAL）",
  "增加真实场景 demo（文件监控、API 健康检查、日志分析）",
  "撰写快速开始指南（5分钟跑起来）",
  "维护 FAQ（常见问题解答）"
]
```

**之后：**
- 每个 Agent 只有 1 个核心任务
- 任务描述明确、边界清晰

**结论：** 已解决

### ✅ 4. 失败原因从"整体超时"变成"可定位的单点问题"

**之前：**
- 错误："Request timeout (60s)"
- 无法定位：不知道是哪个子任务超时

**之后：**
- 如果 Docs_Unified_Writer 超时 → 说明 README.md 合并任务太大
- 如果 Docs_Demo_Writer 超时 → 说明 demo 生成任务太大
- 可以针对性优化

**结论：** 已解决
