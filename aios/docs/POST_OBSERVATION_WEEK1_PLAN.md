# 观察期后第一周行动清单

**版本：** v1.0  
**创建时间：** 2026-03-09  
**执行时间：** 观察期通过后的第一周  
**原则：** 最小行动、受控激活、不增变量

---

## 1. 目标

### 这一周只解决什么

1. **重新激活 1 个 Learning Agent（GitHub_Researcher）**
   - 验证 Learning Agent 的完整调度链
   - 验证写回链和留痕机制
   - 为后续批量激活铺路

2. **查清 coder-dispatcher 3 次失败根因**
   - 这是当前最真实的质量问题
   - 失败率 25%（3/12）需要降下来
   - 找出根因并分类

3. **统一状态模型和数据源口径**
   - 解决 mode vs state.status 不一致
   - 统一 lessons.json / memory/lessons.json / events.jsonl
   - 让健康检查不再误判

### 这一周不解决什么

- ❌ 不开 Skill 自动创建真实闭环
- ❌ 不做 context-matcher-skill
- ❌ 不做主链路大改
- ❌ 不扩新 Agent
- ❌ 不做趋势可视化
- ❌ 不做性能优化

---

## 2. 顺序

按先后写死，避免到时候乱跳：

### Day 1-2: GitHub_Researcher 首次受控激活

**目标：** 验证 Learning Agent 调度链和写回链

**步骤：**
1. 检查 GitHub_Researcher 的配置（agents.json）
2. 手动触发一次（不通过 heartbeat，用独立脚本）
3. 观察执行过程（日志、状态更新）
4. 验证写回（agents.json stats 更新、events.jsonl 记录）
5. 验证留痕（spawn_results.jsonl、task_executions.jsonl）

**输出：**
- GitHub_Researcher 执行报告
- 调度链验证清单
- 写回链验证清单
- 发现的问题清单

### Day 3-4: coder-dispatcher 失败复盘

**目标：** 查清 3 次失败的根因并分类

**步骤：**
1. 读取 coder-dispatcher 的失败记录
2. 逐条分析失败原因（超时、模型错误、任务复杂度、输入问题）
3. 归类失败类型
4. 提出修复建议（不立刻修复，先记录）

**输出：**
- 3 次失败的根因分析报告
- 失败类型分类（超时、模型、任务、输入）
- 修复建议清单（优先级排序）

### Day 5: mode vs state.status 统一

**目标：** 解决状态模型不一致问题

**步骤：**
1. 梳理当前 mode 和 state.status 的使用场景
2. 确定统一标准（用哪个字段、废弃哪个字段）
3. 更新相关代码（agents.json、health_check.py、heartbeat.py）
4. 验证健康检查不再误判

**输出：**
- 状态模型统一方案
- 代码更新清单
- 健康检查验证报告

### Day 6-7: lessons.json / memory/lessons.json / events.jsonl 口径统一

**目标：** 统一数据源，避免多头记录

**步骤：**
1. 梳理当前 3 个文件的用途和内容
2. 确定统一标准（哪个是主数据源、哪个是备份、哪个是历史）
3. 更新相关代码（self_improving_loop.py、experience_learner.py）
4. 验证数据一致性

**输出：**
- 数据源口径统一方案
- 代码更新清单
- 数据一致性验证报告

## 3. 每项验收标准

### GitHub_Researcher 首次受控激活

**通过标准：**
- ✅ 成功触发（spawn_requests.jsonl 有记录）
- ✅ 完成执行（spawn_results.jsonl 有结果）
- ✅ 写回成功（agents.json stats 更新）
- ✅ 留痕完整（events.jsonl 有完整事件链）
- ✅ 无异常报错

**不通过标准：**
- ❌ 触发失败
- ❌ 执行超时
- ❌ 写回失败
- ❌ 留痕缺失
- ❌ 有未处理异常

### coder-dispatcher 失败复盘

**通过标准：**
- ✅ 3 次失败都有根因归类
- ✅ 每个根因都有修复建议
- ✅ 修复建议有优先级排序
- ✅ 输出完整的复盘报告

**不通过标准：**
- ❌ 有失败记录无法归类
- ❌ 根因分析不清晰
- ❌ 没有修复建议

### mode vs state.status 统一

**通过标准：**
- ✅ 确定统一标准（用哪个字段）
- ✅ 相关代码已更新
- ✅ 健康检查不再因字段缺失误判"无活跃 Agent"
- ✅ 所有 Agent 状态字段一致

**不通过标准：**
- ❌ 标准不明确
- ❌ 代码未更新
- ❌ 健康检查仍然误判

### lessons.json / memory/lessons.json / events.jsonl 口径统一

**通过标准：**
- ✅ 确定统一标准（哪个是主数据源）
- ✅ 相关代码已更新
- ✅ 数据一致性验证通过
- ✅ 不再有多头记录

**不通过标准：**
- ❌ 标准不明确
- ❌ 代码未更新
- ❌ 数据不一致

## 4. 不做事项

这一周明确不做：

### ❌ 不开 Skill 自动创建真实闭环
- 原因：观察期刚结束，先验证 Learning Agent 调度链
- 时机：等 Learning Agent 稳定后再做

### ❌ 不做 context-matcher-skill
- 原因：这是 Skill 自动创建的一部分，现在不做
- 时机：等 Skill 自动创建 MVP 启动后再做

### ❌ 不做主链路大改
- 原因：观察期刚结束，先稳定现有链路
- 时机：等问题清单整理完再做

### ❌ 不扩新 Agent
- 原因：现有 30 个 Agent，10 个还没运行过
- 时机：等现有 Agent 稳定后再扩

### ❌ 不做趋势可视化
- 原因：这是 P2 优先级，不是第一周重点
- 时机：等核心问题解决后再做

### ❌ 不做性能优化
- 原因：现在还没有性能瓶颈
- 时机：等系统稳定运行一段时间后再做

## 5. 风险预案

### 风险 1：GitHub_Researcher 激活失败

**可能原因：**
- spawn 机制有问题
- 配置不正确
- 依赖缺失

**应对：**
- 先用独立脚本测试
- 检查 spawn_requests.jsonl 和 spawn_results.jsonl
- 查看详细日志

### 风险 2：coder-dispatcher 失败根因不清晰

**可能原因：**
- 日志不完整
- 错误信息不明确
- 失败场景复杂

**应对：**
- 先分析能分析的
- 不清晰的标记为"待进一步调查"
- 不强求 100% 清晰

### 风险 3：状态模型统一影响现有功能

**可能原因：**
- 字段依赖复杂
- 改动范围大
- 测试不充分

**应对：**
- 先做小范围测试
- 逐步推广
- 保留回滚方案
