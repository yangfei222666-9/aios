# GitHub_Researcher 首次激活验收标准

**版本：** v1.0  
**创建时间：** 2026-03-09  
**执行时间：** 观察期通过后的第一周 Day 1-2  
**目标：** 证明它真的完成了一次可控、可追溯、有价值的学习闭环

---

## 验收标准

### 1. Trigger（触发）

**必须满足：**
- ✅ 任务由手动、定点方式触发（不通过 heartbeat 自动触发）
- ✅ 触发时间可记录（精确到秒）
- ✅ 触发人可记录（谁触发的）
- ✅ 触发原因可记录（为什么触发）

**验证方式：**
- 检查 spawn_requests.jsonl 或等价触发记录
- 确认有明确的触发时间戳
- 确认有触发来源标记

**示例：**
```json
{
  "timestamp": "2026-03-10T10:00:00+08:00",
  "agent": "GitHub_Researcher",
  "trigger_by": "manual",
  "trigger_reason": "首次受控激活验证",
  "task": "搜索和分析 AIOS/Agent/Skill 相关项目"
}
```

---

### 2. Input（输入）

**必须满足：**
- ✅ 输入任务清晰（不是空任务、不是模糊任务）
- ✅ 研究范围明确（有具体主题或项目方向）
- ✅ 至少有指定主题或项目方向

**验证方式：**
- 检查任务描述是否清晰
- 确认研究范围是否明确

**示例：**
```
任务：搜索和分析 GitHub 上的 AIOS/Agent/Skill 相关项目
范围：
- 关键词：AIOS, Agent System, Multi-Agent, Skill, Tool Use
- 时间范围：最近 30 天更新
- Star 数：>100
- 语言：Python
```

**不合格示例：**
```
任务：研究一下 GitHub
范围：不限
```

---

### 3. Execution（执行）

**必须满足：**
- ✅ 有开始时间（精确到秒）
- ✅ 有结束时间（精确到秒）
- ✅ 有耗时记录（秒或分钟）
- ✅ 有执行日志（至少记录关键步骤）
- ✅ 没有中途静默失败（如果失败，必须有明确错误信息）

**验证方式：**
- 检查 spawn_results.jsonl 或等价执行记录
- 确认有完整的时间戳
- 确认有执行日志或关键步骤记录

**示例：**
```json
{
  "agent": "GitHub_Researcher",
  "start_time": "2026-03-10T10:00:00+08:00",
  "end_time": "2026-03-10T10:15:32+08:00",
  "duration_seconds": 932,
  "status": "completed",
  "log": [
    "搜索关键词：AIOS, Agent System",
    "找到 15 个相关项目",
    "分析前 5 个高 Star 项目",
    "生成架构启发 3 条",
    "生成差距判断 2 条",
    "生成改进建议 3 条"
  ]
}
```

**不合格示例：**
```json
{
  "agent": "GitHub_Researcher",
  "status": "completed"
}
```
（缺少时间戳、耗时、日志）

---

### 4. Output（输出）

**必须满足：**
- ✅ 至少输出 3 项：
  1. **1 条架构启发** - 这个项目的核心架构亮点是什么
  2. **1 条与太极OS的差距判断** - 我们的优势和缺口是什么
  3. **1 条可执行改进建议** - 能给太极OS带来什么可执行改进
- ✅ 输出不是空摘要（不是"这个项目很好"）
- ✅ 输出不是泛泛而谈（要有具体细节）

**验证方式：**
- 检查输出内容是否包含 3 项
- 确认每项都有具体内容，不是空话

**示例：**

**架构启发：**
```
项目：hermes-agent
核心亮点：Agent-Curated Memory（Agent 主动创建 Skill）
- Agent 在任务完成后主动判断是否值得抽象为 Skill
- 不是后台自动检测，而是依赖 LLM 判断力
- 三层验证：格式验证 + 安全扫描 + 回滚机制
```

**差距判断：**
```
太极OS vs hermes-agent：
优势：
- 我们有自动检测重复模式的能力
- 我们有 Reality Ledger 记录完整生命周期
劣势：
- 我们缺少 Agent 主动创建 Skill 的能力
- 我们缺少三层验证机制
```

**改进建议：**
```
可执行改进：
在 Skill 自动创建 MVP 中融合两种模式：
1. 自动检测（太极OS）+ Agent 主动创建（hermes-agent）
2. 集成三层验证：格式验证 + 安全扫描 + 回滚机制
3. 优先级：P1，预计 2-3 天完成
```

**不合格示例：**
```
架构启发：这个项目很好
差距判断：我们还有差距
改进建议：继续学习
```

---

### 5. Writeback（写回）

**必须满足：**
- ✅ 结果成功写回指定位置
- ✅ 至少落到以下之一：
  - `memory/YYYY-MM-DD-github-research.md`
  - `agent_system/agent_execution_record.jsonl`
  - `events.jsonl`（如果是标准事件源）
- ✅ 写回内容完整（不是只写了一半）
- ✅ 写回格式正确（可被后续读取和解析）

**验证方式：**
- 检查指定位置是否有新文件或新记录
- 确认内容完整
- 确认格式正确

**示例：**

**写回到 memory/：**
```markdown
# GitHub 研究 - 2026-03-10

## 项目：hermes-agent

### 架构启发
...

### 差距判断
...

### 改进建议
...
```

**写回到 agent_execution_record.jsonl：**
```json
{
  "timestamp": "2026-03-10T10:15:32+08:00",
  "agent": "GitHub_Researcher",
  "task": "搜索和分析 AIOS/Agent/Skill 相关项目",
  "status": "success",
  "output": {
    "insights": [...],
    "gaps": [...],
    "suggestions": [...]
  }
}
```

---

### 6. Outcome（结果）

**必须满足：**
- ✅ 明确标记为：`success` / `partial` / `failed`
- ✅ 如果不是 `success`，要写清原因

**验证方式：**
- 检查执行记录中的 status 字段
- 如果是 `partial` 或 `failed`，确认有原因说明

**示例：**

**成功：**
```json
{
  "status": "success",
  "reason": "完成 3 条架构启发、2 条差距判断、3 条改进建议"
}
```

**部分成功：**
```json
{
  "status": "partial",
  "reason": "只完成 1 条架构启发，因为搜索结果不足"
}
```

**失败：**
```json
{
  "status": "failed",
  "reason": "GitHub API 超时，无法获取项目信息"
}
```

## 失败判定

出现任一即不通过：

### ❌ 触发了但无输出
- 任务触发了，但没有生成任何输出
- 或输出为空

### ❌ 有输出但未写回
- 生成了输出，但没有写回到指定位置
- 或写回失败

### ❌ 写回了但无执行记录
- 写回成功，但没有在 agent_execution_record.jsonl 或 events.jsonl 中留下记录
- 无法追溯执行过程

### ❌ 输出明显脱离任务主题
- 任务是"研究 AIOS 项目"，输出却是"天气预报"
- 输出与任务无关

### ❌ 过程中出现异常但未被记录
- 执行过程中出现错误，但没有记录到日志
- 静默失败

## 通过后的记录

通过后要补 1 条总结，回答以下 3 个问题：

### 1. 这次激活是否证明 Learning Agent 链路可用？

**回答示例：**
```
✅ 是，证明了 Learning Agent 链路可用。

证据：
- 触发机制正常（手动触发成功）
- 执行机制正常（完成任务，有日志）
- 输出机制正常（生成 3 条启发、2 条差距、3 条建议）
- 写回机制正常（成功写回到 memory/）
- 留痕机制正常（agent_execution_record.jsonl 有记录）
```

或：
```
❌ 否，链路有问题。

问题：
- 写回机制失败（无法写入 memory/）
- 留痕机制缺失（agent_execution_record.jsonl 无记录）
```

### 2. 哪个环节最弱？

**回答示例：**
```
最弱环节：写回机制

原因：
- 写回时路径不存在，导致失败
- 没有自动创建目录的逻辑
- 错误信息不明确

改进建议：
- 写回前先检查路径是否存在
- 如果不存在，自动创建
- 失败时记录详细错误信息
```

### 3. 下一次激活前要不要调整？

**回答示例：**
```
✅ 需要调整

调整项：
1. 修复写回机制（自动创建目录）
2. 增强错误日志（记录详细错误信息）
3. 优化输出格式（统一 markdown 格式）

调整后再激活下一个 Learning Agent（Bug_Hunter）。
```

或：
```
❌ 不需要调整

原因：
- 所有环节都正常
- 输出质量符合预期
- 可以直接激活下一个 Learning Agent
```

## 验收流程

### Step 1: 执行前准备
- [ ] 确认观察期已通过
- [ ] 确认 GitHub_Researcher 配置正确
- [ ] 准备触发脚本或命令
- [ ] 准备验收清单

### Step 2: 手动触发
- [ ] 记录触发时间
- [ ] 记录触发原因
- [ ] 执行触发命令
- [ ] 确认触发成功

### Step 3: 观察执行
- [ ] 监控执行日志
- [ ] 记录关键步骤
- [ ] 记录开始时间
- [ ] 记录结束时间
- [ ] 记录耗时

### Step 4: 检查输出
- [ ] 确认有 3 项输出（架构启发、差距判断、改进建议）
- [ ] 确认输出不是空摘要
- [ ] 确认输出不是泛泛而谈
- [ ] 确认输出与任务主题相关

### Step 5: 验证写回
- [ ] 检查 memory/ 是否有新文件
- [ ] 检查 agent_execution_record.jsonl 是否有新记录
- [ ] 确认写回内容完整
- [ ] 确认写回格式正确

### Step 6: 确认结果
- [ ] 确认 status 为 success / partial / failed
- [ ] 如果不是 success，确认有原因说明
- [ ] 记录最终结果

### Step 7: 总结复盘
- [ ] 回答 3 个问题（链路可用？最弱环节？需要调整？）
- [ ] 记录到 POST_OBSERVATION_WEEK1_PLAN.md

## 附录：常见问题

### Q: 如果触发失败怎么办？
A: 检查配置、检查依赖、查看错误日志，修复后重新触发。

### Q: 如果输出质量不高怎么办？
A: 先记录问题，不要立刻修复。等第一周结束后统一复盘。

### Q: 如果写回失败怎么办？
A: 检查路径、检查权限、查看错误日志，修复后重新写回。

### Q: 如果执行超时怎么办？
A: 记录超时时间，分析原因（任务太复杂？网络太慢？），调整后重新执行。

### Q: 如果输出脱离主题怎么办？
A: 标记为失败，分析原因（任务描述不清？模型理解错误？），调整后重新执行。
