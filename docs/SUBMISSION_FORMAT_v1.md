# 提交格式规范 v1

**生效日期：** 2026-03-14

---

## 模板 A：代码正文提交

**核心规则：一次一个函数或一个文件，从第一行到最后一行，不接受摘要，不接受片段。**

提交格式：

```
## 提交：[模块名] / [函数名或文件名]

**文件：** path/to/file.py
**范围：** 函数级 / 文件级
**类型：** 新增 / 修改 / 重构 / 修复

### 正文

[完整代码，从 def / class / 文件第一行到最后一行]
[不要摘要，不要解释，不要过程日志]
[不要"已完成/可运行"结论先行]
```

禁止事项：
- 不贴"关键代码片段"
- 不用变更说明代替正文
- 不在正文前加结论
- 不省略任何行

---

## 模板 B：仓库拆解提交

**核心规则：按 6 个工程镜头拆解，不做泛泛分析。**

提交格式：

```
## 拆解：[项目名/仓库名]

**来源：** [GitHub URL]
**目的：** [为什么要看这个]

### 1. 架构分层
[核心模块、层级关系、入口点]

### 2. 状态流转
[任务/实体的状态机、状态转换条件]

### 3. Task Schema
[任务定义结构、字段、类型约束]

### 4. 错误恢复
[失败处理、重试策略、fallback 机制]

### 5. 审批闸门
[质量门禁、人工确认点、自动化边界]

### 6. 可观察性
[日志、指标、告警、状态可见性]

### 与太极OS的关系
- 可复用：[列出]
- 可借鉴：[列出]
- 不适用：[列出]

### 结论
[一句话定性：值不值得深入、哪部分最有价值]

### 建议下一步
[具体可执行的动作]
```

---

## 模板 C：验证结果提交

**核心规则：不能只写 pass/fail，必须带步骤、实际输出、范围、未覆盖项。**

提交格式：

```
## 验证：[验证对象]

**验证方式：** [脚本/手动/对比]

### 验证范围
[明确本次验证覆盖了什么]

### 验证步骤
1. [具体操作]
2. [具体操作]

### 实际输出
[粘贴真实输出，不是描述]

### 结论
[通过/不通过，基于什么证据]

### 未覆盖项
[本次验证没有覆盖的部分]

### 遗留问题
[如果有的话]
```
