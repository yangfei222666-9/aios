# 太极OS 架构定位与演进方向

**文档版本：** v1.0  
**创建日期：** 2026-03-11  
**文档性质：** 定位文档，非实现规范  

---

## 总判断

太极OS 当前不是"流派未定"，而是已经明确以**宪法驱动**为核心优势，其余三条流派按 **Daemon 化、进程协作、内存主动管理**逐步补齐。

---

## 1. 文档目的

本文档旨在：
1. 明确太极OS 在四条架构流派中的当前定位
2. 识别优势与缺口
3. 排序后续演进优先级
4. 为具体落地提供指导方向

**本文档不是：**
- 实现规范（不直接触发主链路改造）
- 最终架构定稿（允许随系统演进调整）
- 功能清单（不列举所有模块）

---

## 2. 四流派定位

### 流派 1：宪法驱动（Constitutional AI）

**核心理念：** 用规则约束 Agent 行为，让系统在边界内自主决策。

**太极OS 现状：**
- ✅ **最强优势** - 已建立完整的规则体系
  - `lessons.json` - 经验规则库
  - `rules_derived` - 从失败中提炼的规则
  - `AGENTS.md` / `SOUL.md` / `USER.md` - 行为准则
  - Reality Ledger - 真实性验证机制
  - Evolution Score - 改进价值评估
- ✅ 规则可追溯、可更新、可验证
- ✅ 治理框架已初步建立

**当前缺口：**
- ❌ 规则未真正进入执行约束（P0）
  - Agent 执行时不主动加载 `rules_derived`
  - 规则违反无自动拦截
  - 规则冲突无解决机制
- ❌ 规则优先级、依赖关系未建模
- ❌ 规则覆盖率未量化

**改进方向：**
1. **P0** - 让规则真正约束执行
   - Agent 启动时强制加载相关规则
   - 执行前规则预检（rule_guard）
   - 违规自动拦截 + 记录
2. **P1** - 规则关系建模
   - 规则优先级（P0/P1/P2）
   - 规则依赖（requires/conflicts）
   - 规则适用范围（scope）
3. **P2** - 规则覆盖率量化
   - 统计规则覆盖的场景数
   - 识别无规则覆盖的高风险区域

---

### 流派 2：Daemon 化（Long-Running Services）

**核心理念：** 系统作为常驻服务持续运行，而非一次性脚本。

**太极OS 现状：**
- ⚠️ **部分实现** - 有 Daemon 化意识，但未完全落地
  - Memory Server - 常驻进程（端口 7788）
  - Dashboard - 可选常驻（端口 8888）
  - Heartbeat - 定期触发（每 6 小时）
- ⚠️ 依赖 OpenClaw 主会话心跳，非独立 Daemon
- ❌ 无进程管理、无自动重启、无健康检查

**当前缺口：**
- ❌ 非真正的 Daemon（P1）
  - 依赖外部触发（OpenClaw 心跳）
  - 无独立进程管理
  - 无自动重启机制
- ❌ 无 IPC 机制
  - Agent 间通信依赖文件系统
  - 无消息队列、无事件总线
- ❌ 无服务注册与发现

**改进方向：**
1. **P1** - 真 Daemon 化
   - 独立进程管理（systemd / Windows Service）
   - 自动重启 + 健康检查
   - 进程间通信（IPC）
2. **P2** - 服务化架构
   - 服务注册与发现
   - 负载均衡（多 Agent 实例）
   - 故障隔离与降级

---

### 流派 3：进程管理（Multi-Agent Orchestration）

**核心理念：** 多个 Agent 协作完成复杂任务，需要调度、协调、资源管理。

**太极OS 现状：**
- ⚠️ **基础设施已有，协作能力弱**
  - Task Queue - 任务队列（`task_queue.jsonl`）
  - Agent Registry - Agent 注册表（`agents.json`）
  - Spawn 机制 - 子 Agent 创建（`sessions_spawn`）
- ⚠️ 调度逻辑简单（FIFO + 优先级）
- ❌ 无真正的多 Agent 协作（无依赖、无并行、无冲突检测）

**当前缺口：**
- ❌ 无 Agent 间协作机制（P1）
  - 无任务依赖（Task A 完成后才能执行 Task B）
  - 无并行执行（多个 Agent 同时工作）
  - 无资源竞争检测（两个 Agent 同时修改同一文件）
- ❌ 调度策略单一
  - 只有 FIFO + 优先级
  - 无负载均衡、无公平调度
- ❌ 无 Agent 生命周期管理
  - 无超时控制、无资源回收
  - 无僵尸进程检测

**改进方向：**
1. **P1** - 多 Agent 协作
   - 任务依赖图（DAG）
   - 并行执行 + 资源锁
   - 冲突检测 + 自动回滚
2. **P2** - 高级调度
   - 负载均衡（按 Agent 负载分配任务）
   - 公平调度（防止饥饿）
   - 优先级抢占（高优先级任务插队）
3. **P2** - 生命周期管理
   - 超时控制 + 自动终止
   - 资源回收（内存、文件句柄）
   - 僵尸进程清理

---

### 流派 4：内存分页（Memory Management）

**核心理念：** 主动管理记忆，按需加载，避免上下文爆炸。

**太极OS 现状：**
- ❌ **最弱环节** - 记忆管理基本是被动的
  - Memory Server - 提供检索能力（embedding + LanceDB）
  - `MEMORY.md` - 长期记忆（手动维护）
  - `memory/YYYY-MM-DD.md` - 日志（手动写入）
- ❌ 无主动压缩、无自动归档、无遗忘机制
- ❌ 无上下文预算管理

**当前缺口：**
- ❌ 无主动内存管理（P2）
  - 无自动压缩（旧记忆摘要化）
  - 无自动归档（冷数据迁移）
  - 无遗忘机制（过期数据清理）
- ❌ 无上下文预算
  - 无 token 预算管理
  - 无上下文优先级（哪些记忆必须加载）
- ❌ 无渐进式加载
  - 无 Level 0/1/2 加载策略（参考 hermes-agent）

**改进方向：**
1. **P2** - 主动内存管理
   - 自动压缩（定期摘要旧记忆）
   - 自动归档（冷数据迁移到归档区）
   - 遗忘机制（过期数据自动清理）
2. **P2** - 上下文预算
   - Token 预算管理（限制每次加载的记忆量）
   - 上下文优先级（核心记忆 > 相关记忆 > 背景记忆）
3. **P3** - 渐进式加载
   - Level 0：索引（只加载标题和摘要）
   - Level 1：完整内容（按需加载详细内容）
   - Level 2：支持文件（按需加载附件）

## 3. 太极OS 当前优势

### 3.1 宪法驱动（核心优势）

- **完整的规则体系** - `lessons.json` + `rules_derived` + 行为准则
- **Reality Ledger** - 真实性验证机制，防止假运行
- **Evolution Score** - 改进价值评估，防止无效改进
- **治理框架** - 验收模板、状态模型、优先级排序

### 3.2 可观测性

- **完整的执行记录** - `agent_execution_record.jsonl`
- **任务队列可追溯** - `task_queue.jsonl` + `task_executions.jsonl`
- **健康检查机制** - `aios-health-check` + `aios-health-monitor`
- **备份与恢复** - MRS 标准 + 定期演练

### 3.3 学习能力

- **失败沉淀机制** - 从失败中提炼规则
- **Skill 自动创建** - 从重复模式中生成能力
- **Self-Improving Loop** - 自动改进机制

### 3.4 方法论

- **太极哲学** - 阴阳平衡、动态演化、万物归一
- **观察期原则** - 先观察、再验证、最后接入
- **最小验收模板** - 6 项验收标准（Trigger/Input/Execution/Output/Writeback/Outcome）
- **分层推进** - Phase 1（学习型）→ Phase 2（执行型）→ Phase 3（自主型）

## 4. 当前缺口

### P0 缺口（影响系统可信度）

1. **规则未真正进入执行约束**
   - Agent 执行时不主动加载规则
   - 规则违反无自动拦截
   - 这让宪法驱动的优势无法发挥

2. **Agent 状态模型失真**
   - 注册 ≠ 可执行 ≠ 已验证
   - 大量 Agent 处于 `registered / not executable / not validated` 状态
   - 这让 Dashboard、健康检查、日报都失真

3. **selflearn-state.json 缺失**
   - 学习系统是否在运行不可判定
   - 核心状态断点

4. **lesson 系统只记失败，不产规则**
   - 学习闭环断裂
   - 9 条 lesson 中 6 条 `pending`，内容空

### P1 缺口（影响系统扩展性）

1. **非真正的 Daemon**
   - 依赖外部触发（OpenClaw 心跳）
   - 无独立进程管理、无自动重启

2. **无 Agent 间协作机制**
   - 无任务依赖、无并行执行、无冲突检测
   - 限制了复杂任务的处理能力

3. **无 IPC 机制**
   - Agent 间通信依赖文件系统
   - 效率低、可靠性差

### P2 缺口（影响系统长期运行）

1. **无主动内存管理**
   - 无自动压缩、无自动归档、无遗忘机制
   - 长期运行会导致上下文爆炸

2. **无上下文预算管理**
   - 无 token 预算、无上下文优先级
   - 无法控制成本

3. **调度策略单一**
   - 只有 FIFO + 优先级
   - 无负载均衡、无公平调度

## 5. 优先级排序

### P0（立即处理）

1. **让规则真正约束执行**
   - Agent 启动时强制加载相关规则
   - 执行前规则预检（rule_guard）
   - 违规自动拦截 + 记录

2. **统一状态模型**
   - 定义统一状态词表（registered/executable/validated/production-ready/stable/archived）
   - 更新所有 Agent 状态
   - 同步到 Dashboard

3. **恢复 selflearn-state.json**
   - 最小化恢复（last_run/last_success/activated_agents/pending_lessons/rules_derived_count）
   - 让学习链可观测

4. **打通 lesson→rule 最小闭环**
   - 从 1 条真实 lesson 提炼 1 条可复用规则
   - 写回规则区
   - 证明"能学到东西"

### P1（近期规划）

1. **真 Daemon 化**
   - 独立进程管理（systemd / Windows Service）
   - 自动重启 + 健康检查
   - 进程间通信（IPC）

2. **多 Agent 协作**
   - 任务依赖图（DAG）
   - 并行执行 + 资源锁
   - 冲突检测 + 自动回滚

3. **队列契约收口**
   - queued 去重
   - failed 标记
   - unknown task_id 处理

### P2（长期优化）

1. **主动内存管理**
   - 自动压缩（定期摘要旧记忆）
   - 自动归档（冷数据迁移）
   - 遗忘机制（过期数据清理）

2. **上下文预算管理**
   - Token 预算管理
   - 上下文优先级
   - 渐进式加载（Level 0/1/2）

3. **高级调度**
   - 负载均衡
   - 公平调度
   - 优先级抢占

## 6. 边界说明

### 本文档的作用

- **指导方向** - 明确太极OS 的架构定位和演进方向
- **优先级排序** - 帮助决策"先做什么，后做什么"
- **差距识别** - 清晰认识优势与缺口

### 本文档不做的事

- **不直接触发主链路改造** - 具体落地仍按观察、验收、渐进接入来做
- **不是实现规范** - 不规定具体实现细节
- **不是最终定稿** - 允许随系统演进调整

### 落地原则

1. **观察期原则** - 先观察、再验证、最后接入
2. **最小验收模板** - 每个改进必须通过 6 项验收
3. **分层推进** - Phase 1（学习型）→ Phase 2（执行型）→ Phase 3（自主型）
4. **优先级驱动** - P0 > P1 > P2，不跳级
