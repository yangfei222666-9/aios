# 开源项目工程拆解模板

**用途：** 对开源项目做工程借鉴判断，而非营销分析

**核心原则：** 不复读 README，不写空话，必须引用真实代码/结构，区分工程事实和营销叙事

---

## 任务目标

从工程实现角度拆出：
- 这个项目哪些骨架值得借鉴
- 哪些不能抄
- 哪些适合太极OS当前阶段

---

## 你必须做的事

1. **阅读仓库真实结构**
2. **优先看：**
   - orchestration / scheduler
   - task / state
   - approval / retry / pending / blocked
   - memory / trace / audit / dashboard
3. **识别：**
   - 真实骨架
   - 营销包装
   - 对太极OS当前阶段的借鉴价值

---

## 你不要做的事

- ❌ 不要复读 README
- ❌ 不要只看首页介绍就下结论
- ❌ 不要写"很先进""很有前景"这种空话
- ❌ 不要被高概念口号带节奏
- ❌ 不要跳过仓库结构直接输出感想

---

## 输出格式（9 个部分）

### 1. 项目定位判断

- **README/宣传语怎么定义它**
- **你判断它本质上是什么**
- **一句话定性**

### 2. 架构分层

- 是否有 orchestrator / worker / reviewer / memory / tool layer
- 各层职责
- 哪层是真核心

### 3. 任务系统

- task schema 长什么样
- 核心字段有哪些
- 任务如何流转
- 是否有 review / pending / blocked / approval

### 4. 状态机与持久化

- 状态存在哪
- 是否支持恢复 / 续跑
- 是否有 history / audit / trace

### 5. 错误处理与安全

- 有无错误分类
- 有无 pending / blocked / manual intervention
- 有无权限闸门
- 高风险动作是否需要人审批

### 6. 可观察性

- 用户是否能追踪"谁做了什么"
- 是否有 timeline / dashboard / trace / audit log

### 7. 自治真实性评估

- 人类实际上还在哪些环节
- 它到底接近自治，还是高概念包装
- 无人值守能持续多久（按你能看到的信息推断）

### 8. 对太极OS的借鉴评估

分三栏：

**可直接借**
- 现在就能吸收的

**可改造借**
- 以后可以吸收，但要改造的

**暂时别借**
- 当前阶段不适合的
- 风险高的
- 叙事大于工程的

### 9. 最终结论

- **一句话结论**
- **最值得借的 3 点**
- **最不能抄的 3 点**
- **对太极OS的下一步建议**

---

## 额外硬要求

### 1. 必须引用仓库里的真实结构

例如：
- 目录名
- 文件名
- 模块名
- 字段名
- 状态名

### 2. 必须区分三层

- **工程事实：** 代码里真实存在的
- **README/宣传语：** 项目自己怎么说的
- **你的判断：** 基于事实的工程评估

### 3. 重点回答核心问题

"这个项目的哪些工程骨架值得太极OS借鉴，而不是被它的口号误导？"

---

## 交付标准

我要的不是"看起来懂了"，而是：

✅ 能看出你真的读了仓库  
✅ 能区分工程和叙事  
✅ 能落到太极OS能用什么

---

## 最后一句

**不要先夸项目，先拆项目。**

目标是工程借鉴判断，不是项目宣传。
